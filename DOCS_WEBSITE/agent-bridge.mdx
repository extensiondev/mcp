If you have ever tried to point an AI assistant — or Playwright, or the Chrome DevTools Protocol (CDP) — at a browser extension, you already know the wall: those tools see **pages**. Your extension lives somewhere else. Its logic runs in a service worker, its content scripts run in an isolated world, its UI lives in popup, options, and side-panel documents. The interesting data is exactly where page-centric tools can't reach.

The **agent bridge** closes that gap. When you run `extension dev`, Extension.js loads a privileged companion alongside your extension and opens a local channel that lets a human (a sidebar), a script (a file), or an AI agent (a live stream) **observe and act on every extension surface** — on Chrome *and* Firefox.

<Note>
The agent bridge is a **free, local, open-source** part of Extension.js. It runs entirely on `127.0.0.1`. No account, no cloud, no data leaves your machine. The only paid surface in the platform is hosted [distribution](/docs/ai/publish), which is unrelated to the bridge.
</Note>

## What it unlocks

| You want to… | Without the bridge | With the bridge |
| --- | --- | --- |
| See logs from the service worker **and** content script in one timeline | two DevTools windows, eyeballed timestamps | one ordered stream ([Agent logs](/docs/ai/logs)) |
| Inspect your options page DOM on Firefox | unsupported by CDP | works ([Inspect and act](/docs/ai/inspect-and-act)) |
| Call a background function from your terminal | wire a temp button, reload | `extension eval` |
| Let an assistant debug a failing message | paste console screenshots | the agent reads structured diagnostics directly |

## How it fits together

```
your extension  ──┐
                  ├─▶  Extension.js companion  ──▶  local control channel (127.0.0.1)
your dev build  ──┘        (auto-loaded)                 │
                                                         ├─▶  sidebar UI        (you, live)
                                                         ├─▶  *.ndjson files    (any script/agent)
                                                         └─▶  MCP tools          (your AI assistant)
```

You never opt into a "mode." You run `extension dev` the way you always have. The companion loads automatically, the files are always written, and the channel is there if a tool wants it. Three ways to consume **one** data plane:

## Two trust levels

The bridge is deliberately split so that *reading* is effortless and *acting* is deliberate.

| Capability | What it does | Gate |
| --- | --- | --- |
| **Observe** — logs, diagnostics, DOM inspection | read-only view of what your extension is doing | none (localhost) |
| **Bounded act** — `storage`, `reload`, `open`, `tabs` | safe, scoped operations | `--allow-control` |
| **Eval** — run arbitrary code in a context | maximum power, maximum risk | `--allow-eval` + a per-session token |

Observation is on by default. Acting is something you turn on for a session, on purpose. See [Inspect and act](/docs/ai/inspect-and-act#safety) for why `eval` is gated the way it is.

## Honest limits

We would rather tell you where a tool stops than let it lie to you.

- **Closed shadow roots:** the default (companion-based) inspector reads open shadow roots. Closed ones need `--deep-dom` (Chromium only).
- **Network:** the bridge streams console output and structured diagnostics, not your extension's outbound `fetch`/XHR. Use DevTools for network-level debugging.
- **Dropped events under flood:** if you log thousands of events per second, the stream reports exactly how many it coalesced or dropped — it never silently loses data and pretends otherwise.
