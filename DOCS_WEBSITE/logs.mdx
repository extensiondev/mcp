Debugging an extension usually means juggling consoles: one for the page, one for the service worker, one for the popup. The clocks don't line up and you reconstruct the story by hand.

Agent logs collapse that into **one ordered stream** of everything your extension does, across every context, available three ways: a terminal command, an [NDJSON](https://github.com/ndjson/ndjson-spec) file, and a live [MCP](/docs/ai/mcp) tool. It is part of the free, local [agent bridge](/docs/ai/agent-bridge).

## When to use

- Following a bug that crosses contexts (content script → service worker → popup).
- Letting an AI assistant read what your extension is doing without screenshots.
- Capturing a session to a file for CI or later inspection.

## The file is always there

While `extension dev` runs, Extension.js writes every log line to:

```
dist/extension-js/<browser>/logs.ndjson
```

It is **on by default** and rotates automatically (caps at 8 MB / 50k lines, keeps three generations). A fresh dev run starts a clean file; the previous one rolls to `logs.1.ndjson`. The first line is always a header that records the schema version and `runId`.

Because it's a plain file, any tool can read it with no setup:

<CodeGroup>

```bash Tail everything
tail -f dist/extension-js/chromium/logs.ndjson
```

```bash Only the service worker + content script
tail -f dist/extension-js/chromium/logs.ndjson \
  | jq 'select(.context=="background" or .context=="content")'
```

```bash Only diagnostics that need attention
tail -f dist/extension-js/chromium/logs.ndjson \
  | jq 'select(.eventType=="dx.signal" and .status=="fail")'
```

</CodeGroup>

## The `logs` command

For a friendlier view, use the command. It reads the same data and, with `--follow`, subscribes to the live channel.

<CodeGroup>

```bash npm
extension logs [path] [options]
```

```bash pnpm
extension logs [path] [options]
```

```bash yarn
extension logs [path] [options]
```

```bash bun
extension logs [path] [options]
```

```bash bun
extension logs [path] [options]
```

</CodeGroup>

By default it prints the current session's logs and exits — a bounded snapshot, ideal for "what just happened." Add `--follow` to watch live.

### Flags

| Flag | What it does | Default |
| --- | --- | --- |
| `--follow` | Subscribe to the live stream instead of dumping and exiting. | dump + exit |
| `--context <list>` | Filter contexts (`background`, `content`, `page`, `sidebar`, `popup`, `options`, `devtools`). | all |
| `--level <off…all>` | Minimum level (`error`, `warn`, `info`, `debug`, `trace`, `all`). | `all` |
| `--signals-only` | Show only structured diagnostics (`dx.signal`) and skip plain console noise. | off |
| `--since <seq\|iso>` | Resume from a sequence number or timestamp. | start of session |
| `--url <glob\|substring>` | Only events whose url/hostname matches (glob with `*`, or a plain substring). | all |
| `--tab <id>` | Only events from this tab id. | all |
| `--output <pretty\|json\|ndjson>` | Output format. | `pretty` for a TTY, `ndjson` when piped |
| `--browser <target>` | Which `dist/extension-js/<browser>/` to read. | `chromium` |

## What a log line looks like

Every line is the same shape on the wire, in the file, and in the sidebar. The fields that matter most for debugging:

```jsonc
{
  "seq": 2,                          // monotonic order — use this, not the clock
  "level": "error",
  "context": "content",              // which surface emitted it
  "messageParts": ["sendMessage failed", { "err": "Could not establish connection" }],
  "eventType": "dx.signal",          // a structured diagnostic, not a plain log
  "code": "RUNTIME_NO_RECEIVER",
  "status": "fail",
  "remediation": "The service worker may be asleep. Register the listener at top level.",
  "url": "https://shop.example/checkout",
  "tabId": 7
}
```

<Tip>
`dx.signal` events carry a **`remediation`** — the fix travels with the diagnostic. This is what makes the stream useful to an AI assistant: it doesn't have to guess the cause from raw console text, it reads the suggested fix directly.
</Tip>

Order by `seq`, not `timestamp`: it's a single monotonic counter assigned as events arrive, so interleaving contexts is exact even when their clocks drift. If the stream ever has to drop events under heavy load, it emits a `gap` marker and you'll see `seq` jump — you always know when you're missing data.

## For AI assistants

With the [MCP server](/docs/ai/mcp) installed, your assistant calls `extension_logs` to get the same stream as structured results (bounded by default, so it won't flood its own context). A typical ask:

> "Tail the background logs and tell me why `sendMessage` is failing."

The assistant reads the `dx.signal` with `code: RUNTIME_NO_RECEIVER` and its `remediation`, and proposes the fix — no screenshots, no copy-paste.

## Best practices

- **Reach for `--signals-only` first.** Most of the time you want "what's wrong," not every `console.log`.
- **Order by `seq`.** Timestamps are for humans; `seq` is the source of truth for ordering.
- **Commit nothing from `dist/`.** `logs.ndjson` is a per-session artifact and rotates on its own.
